You are in the final stretch of the Hello World tutorial. You've already learned a ton. Good news is that we are going to continue to learn! The new concept that I want to introduce is the B.R.E.A.D. Application. In some circles they call this the C.R.U.D. application. I don't like to call things that I build CRUD.

B.R.E.A.D. stands for:

Browse
Read
Edit
Add
Delete

And it relates to all of the functions that a database application should be able to do. The rest of the tutorial will be building out the BREAD capabilities of the "What is your favorite color" Application.

We will start with the Browse function. Paste the following in your hello.py file.

# Browse ======================================================================
@app.route("/browse")
def browse():

   # Let's query the database and pull the entries
   c.execute('SELECT rowid, name, color FROM favorite_color')
   data = c.fetchall()

   # Send these to the template
   return render_template("browse.html", data=data)

There are two unique things about the Browse function. First, is the rowid column in the SELECT statement. SQLite automatically provides an incrementing id column called rowid. This will be very useful in the later BREAD methods. Second, is the `data=data` in the render_template function. We are passing the results of the query into the template and naming the data to `data`. Now let's build the template.

{% include 'header.html' %}

One of the tenants of programming is DRY (Don't Repeat Yourself). By including a header.html file we can setup our <head> tag in a single file but get the benefit of it everywhere. We will create the header file in a moment.

The HTML <table> tag is the best way to store what we call tabular data. basically, think of it as antyhing that would look good formated like a spreadsheet. It may not make as much sense for this application but I wanted to show you how to do it and help you get familiar with the <table> tags.

<thead> - Table Header. Define the first row of a table (normally to provide column names
<tr> - Table Row. Similar to the <div class='row'> but for the table
<th> - Table Heading. This tag is used to define a column heading
<tbody> - Table Body. This will be the start of the data rows.
<td> - Table Data. This is the data in a column

Tables take a little bit of practice to get the hang of. Just like the Bootstrap Grid.

Now that we can add, let's setup a way to delete them. This way as we are testing we don't get overwhelmed.

Add the delete() function below, after your save() in the hello.py file.

# Delete =======================================================================
@app.route("/delete/")
def delete(id):

   # Delete without confirming anything
   c.execute('DELETE FROM favorite_color WHERE rowid=?', (id,))

   # Commit causes the system to write to the database file
   conn.commit()

   # We want to take the user back to the browse screen
   return redirect(url_for('browse'))

This delete is a little dangerous because it doesn't confirm anything. It just removes it. Might be smart to add a JavaScript confirmation dialog to help the user avoid accidentally clicking the button. For now, they are on their own.

We can add and delete things but what if we want to change our answer? We need to create the edit mode.

We will need two functions for this. The edit which will display the page loaded with existing values and the update function to save the updated values in the database.

Let's start with the edit function.

# Edit =========================================================================
@app.route("/edit/<id>")
def edit(id):

   data = {}
   c.execute('SELECT rowid, name, color FROM favorite_color WHERE rowid=?', (id,))
   data = c.fetchone()

   return render_template("edit.html", data=data)

You've seen most of this before. One difference is the variable passed on the URL. The @app.route has an <id> being passed in. That lines up with the id being passed in the edit(id) function definition. This helps us know which item we are editting. Another difference is the c.fetchone() instead of the c.fetchall().

Now we can add the update() function.

# Update =======================================================================
@app.route("/update", methods=["POST"])
def update():

   # We want to save this response to the database
   c.execute('UPDATE favorite_color SET name = ?, color=? WHERE rowid = ?',
             (request.form['name'], request.form['color'], request.form['id']))

   # Commit causes the system to write to the database file
   conn.commit()

   return redirect(url_for('browse'))

The main new thing in this function is the SQL UPDATE statement. The format is different than an INSERT but the ? variable substitution idea is the same. The 3 ?'s are replaced by the variables in the tuple. We are pulling those variables from the HTML form via the Flask request object.

Our program has one fatal security flaw that we need to fix. When we added the login page we didn't put in any logic to force the user to go to the login screen if they aren't logged in. We need to add that layer of security!

To see what I mean click the `logout` button then visit http://{{request.cookies.vm_name}}/browse

Oops! Luckily Python Flask offers a simple solution to this problem. There is an @app.before_request function that will allow you to run code before any other function on every single browser request.

Let's add the following check_session() function to check if the user has a session and if not we can send them to the login screen.

# Check the session to make sure they are logged in ============================
@app.before_request
def check_session():

   # if we don't see the session then we need to send them to the login
   if 'username' not in session or not session['username']:

      # We also don't want to send them to the login screen if they are looking for
      # Something insecure (i.e. static js files or if they are in the process of
      # Logging in.
      exclude_paths = ['/', '/login']
      if not re.search('static', request.path) and request.path not in exclude_paths:

         # We need to go back to the index page
         return redirect(url_for("index"))

Let's look at each section of this code.

# Check the session to make sure they are logged in ============================
@app.before_request
def check_session():

This section starts with the @app.before_request decorator that tells Flask to run this function before every request. Now it's important to understand that this will run before EVERY request, including the ones to static files like bootstrap.min.css. Since we don't want users to have to login to get CSS or JS files, we will have to handle that in the code below.

# if we don't see the session then we need to send them to the login
if 'username' not in session or not session['username']:

The session object is the magic on Flask sessions. We look first to see if there is any value called username and then second to see if the value is populated with something. In Python the test `if variable` will test for the variable to have any value. at this point we are letting sessions live until the browser window is closed. If you needed to limit a session to a specific amount of time you could add that to the session object. Perhaps like session['expire_time'].

# We also don't want to send them to the login screen if they are looking for
# Something insecure (i.e. static js files or if they are in the process of
# Logging in.
exclude_paths = ['/', '/login']
if not re.search('static', request.path) and request.path not in exclude_paths:

This section above allows us to ignore the pages in exclude_paths as well as any URL that contains the word static. The re.search is a feature in Python to look for that pattern anywhere in a string. In this case the url `/static/css/bootstrap.min.css` will match and we won't redirect them to the login page when they request the Bootstrap CSS.

# We need to go back to the index page
return redirect(url_for("index"))
